# 需求理解与假设说明 + 大模型调用思路

> 适用系统：V3 运单全流程管理系统（异常工单 / 品控 / 分级审批 / 执行联动）
> 关联系统：V2 录单解析系统（出库单 → 运单结构化数据，含大模型辅助解析）
> 文档定位：对需求的理解、在需求留白处所做的假设与依据、以及大模型（LLM）在两系统间的调用思路。

---

## 一、需求理解

### 1.1 系统定位

V3 是一套**运单异常全流程管理系统**，目标是把"仓库扫描 → 发现异常 → 上报工单 → 分级审批 → 执行赔付/库存联动"这条链路数字化、可追溯、可审计。它**不拥有运单主数据**，运单真实性与 SKU 归属以 V2 系统为唯一权威来源，通过 HTTP 接口实时校验（绝不直连 V2 数据库）。

V2 是一套**录单解析系统**：把 Excel / Word / PDF 出库单解析为结构化运单（含外部单号、收货人、SKU 明细等），并对外暴露 `/api/v3/*` 兼容接口供 V3 调用。其中**大模型用于辅助生成"解析规则"**（见第三章）。

### 1.2 业务链路（端到端）

```
仓库扫描枪 ──POST /api/scan──▶ V3
                               │ 1. 实时调用 V2 校验运单存在性 + SKU 归属
                               │ 2. 品控规则引擎判定 (QC)
                               ▼
                         扫描记录(ScanRecord)
                               │ 命中品控规则?
                               ▼
                         异常工单(ExceptionTicket, QC/LOGISTICS)
                               │ 分级审批引擎
                               ▼
                   一级/二级审批(ApprovalRecord) ──通过──▶ 执行联动
                                                       ├─ 赔付记录(CompensationRecord)
                                                       └─ 库存调整(InventoryRecord)
```

### 1.3 功能模块（对应评分模块 0–6）

| 模块 | 能力 | 关键设计点 |
|------|------|-----------|
| 模块零 扫描/品控 | 模拟扫描枪录入、SKU 归属实时校验、5 类品控规则、批次暂扣 QC_HOLD、幂等 | 真实性校验走真实 V2 接口，不拿过期缓存凑数 |
| 模块一 异常上报 | 物流异常手工上报、真实性校验、重复上报检测、5 类异常 | 单租户假设（见假设⑨外补充） |
| 模块二 分级审批 | 一/二级审批、并发乐观锁、离职兜底、超时自动流转、权限边界、幂等 | 业务核心考点 |
| 模块三 执行联动 | 赔付生成（向客户/向供应商）、库存自动调整、事务一致性 | 审批+执行同事务，外键可追溯 |
| 模块四 工单追踪 | 多维筛选、分页、详情、审批审计、超时醒目提示 | — |
| 模块五 跨系统接口 | V2 接口监控、调用统计、日志、数据新鲜度标注、降级/重试 | 考点5 接口可调试性 |
| 模块六 假设说明 | 九项假设 + 待澄清问题 + V2 契约 + 多租户说明 | 本文件即其书面化 |

### 1.4 核心考点理解（考试/评审视角）

- **真实性校验（考点5）**：异常处理必须基于真实存在的运单。V3 在"上报/扫描"时实时调 V2 校验，而非信任本地缓存；仅当 V2 整体不可达才降级到本地快照并明确标注"数据可能非最新"。
- **降级边界（关键）**：V2 明确返回"不存在"（404 / `success:false`）是业务结论，**绝不**回退本地缓存把假运单校验成真；V2 不可达（超时/5xx/网络错）才降级。
- **一致性（考点）**：审批通过与赔付/库存调整在同一数据库事务内完成，靠外键（approvalRecordId / ticketId）保证端到端可追溯。

## 二、假设说明（需求留白处的决策与依据）

以下内容在需求文档 3.4 业务规则留白处做了明确假设，均**可系统配置**（`system_config` 表），便于上线后调整，不硬编码。

### ① 分级审批金额阈值
- **假设**：一级阈值 5,000 元，二级阈值 20,000 元；比对用的"异常金额"由**上报人在 V3 录入**。
- **依据**：参考国内企业常见分级。关键说明——**V2 解析产出的运单模型不含金额字段**（解析源为出库单文本，无货值信息），故 V3 不依赖 V2 返回金额，而是用 `ExceptionTicket.amount` 作为阈值比对与赔付计算的统一依据。这样既驱动可配置的金额分级，又不强制改造 V2 解析引擎。

### ② 审批超时时长
- **假设**：一级超时 1 小时（3600s），二级超时 2 小时（7200s）。
- **依据**：一级通常是部门负责人响应快；二级需跨部门协调给更长时间。超时后升级/驳回策略可配。

### ③ 重提次数上限
- **假设**：驳回后允许重新提交最多 **3 次**，超过自动关闭需重新上报。
- **依据**：平衡用户体验与风险控制，避免无限循环。

### ④ 物流异常类型映射
- **假设**：

  | 异常类型 | 处理 | 是否赔付 |
  |---------|------|---------|
  | 丢件/破损 | 赔付客户 + 重发/退货入库 | 是 |
  | 拒收 | 重新发货 | 否 |
  | 超时未签收 | 赔付客户 + 升级处理 | 是 |
  | 地址错误 | 重新发货 | 否 |

- **依据**：承运方责任（丢件/破损）需赔付；客户原因（拒收）不赔；下单失误（地址错）只重发。

### ⑤ 角色权限划分
- **假设**：操作员（扫描/上报）＜ 品控主管（+ 品控复核/快速放行）＜ 一级审批人 ＜ 二级审批人；**自批自核禁止**。
- **依据**：职责分离原则，避免权限过大。

### ⑥ V2 数据同步频率与一致性策略
- **假设**：**实时校验**（上报/扫描时实时调 V2）+ **本地只读快照缓存**（标注 `syncedAt`）。
- **依据**：异常处理必须基于真实运单，定时同步会造成"凭过期缓存把不存在的运单校验成存在"的伪对接。
- **降级边界（已落实在代码）**：
  - V2 明确返回"不存在" → V3 直接判不存在，**不**回退本地缓存；
  - V2 整体不可达 → 降级本地快照，前端标注"数据可能非最新"；
  - 本地快照只读，不回写运单状态，运单状态以 V2 为准；
  - 每次跨系统调用生成可追踪 `Request ID` 并写入 `SyncLog`（接口名/入参摘要/状态码/耗时/错误），可按 ID 还原调用链。

### ⑦ 品控暂扣超时时长
- **假设**：**15 分钟（900s）**，独立于审批超时。
- **依据**：暂扣=货物锁仓产生仓储成本，应比审批更短，避免无限压货。

### ⑧ 品控规则触发阈值
- **假设**：数量差异 ≥10%、破损等级 ≥3 级、规格偏差 >5%、标签缺失/明显错误、批次过期/临期/召回 立即触发。
- **依据**：仓储行业常见阈值；全部可配置。

### ⑨ 品控主管权限边界
- **假设**：品控主管 ≠ 审批人，不能审批自己触发的工单；快速放行绕过完整审批但**必须填复核原因留痕**。
- **依据**：专业判断权提高效率，但权限与审批分离避免利益冲突，留痕保证可追溯。

### ⑩ 多租户假设（补充说明）
- **当前假设**：**单租户**。若要支持多租户，需在所有业务表加 `tenant_id`、查询强制带 `tenant_id`、权限模型与接口鉴权携带租户信息，禁止跨租户访问。

### ⑪ V2 接口契约与"老系统二开"意识
- **契约（V3 → V2，均 HTTP，绝不直连库）**：
  - `POST /api/v3/orders/validate` 校验运单存在并返详情
  - `POST /api/v3/orders/{orderId}/validate-sku` 校验 SKU 归属
  - `POST /api/v3/orders/sync` / `/batch` 条件查询 / 批量获取（快照初始化与增量同步）
- **鉴权**：所有 V2 对外接口要求 `X-API-Key`，与 V2 本地 `V2_API_KEY` 一致才放行，否则 401（**不是裸奔接口**）。
- **二开意识**：
  - 新增接口统一挂在 `/api/v3/*` 版本前缀下，与 V2 既有 `/api/parse`、`/api/orders` 完全隔离；
  - 字段演进兼容：V3 以 `number` 接收金额并对字段缺失兜底为 `null`，不因 V2 增删字段整体报错；
  - 以 `success` 字段表达业务结果，区分"明确不存在(404)"与"服务不可用(超时/5xx)"。

### 待产品经理澄清的问题（节选）
1. 品控暂扣超时后是自动升级还是自动驳回？（当前设计：升级到二级审批）
2. 同一运单是否允许并存不同类型异常工单？（当前：允许）
3. 降级处理的具体折扣比例？（默认 8 折，可配）
4. 多商户/多仓库数据隔离如何实现？（当前单租户）
5. V2 接口鉴权机制？（当前简单 API Key，生产建议 JWT/OAuth）
6. 品控规则是否支持组合触发？（当前支持，取最高严重度）
7. 库存是否独立于 V2/WMS？（当前 V3 自有库存记录，经接口与 V2 同步）
8. 批量测试数据 seed 脚本是否提供？（当前需另行开发）

> **部署注意（实测结论）**：Vercel 不读取仓库内 `.env`，V3 部署环境必须显式配置 `V2_API_URL`、`V2_API_KEY`（V3 代码对二者已做兜底默认值，但生产应以环境变量为准）；且 V2 部署须关闭/排除其 **Deployment Protection（预览保护）**，否则 V3 服务端调用会被 Vercel Auth 拦截返回 401。

## 三、大模型（LLM）调用思路

### 3.1 当前大模型落地点（在 V2 系统内）

大模型当前**实际用于 V2 的"解析规则辅助生成"**，由 `src/lib/ai.ts` 的 `generateRuleWithAI(filePreview, fileName)` 实现：

- **触发场景**：用户上传 Excel/Word/PDF 出库单后，系统用文件内容预览（前 N 行文本）请求大模型，自动产出一份通用的 `ParseRuleConfig` 解析规则，把非结构化出库单解析为结构化运单数据。
- **模型与参数**（均环境变量驱动，便于切换）：
  - `OPENAI_API_KEY`：必填，缺省/占位符 `sk-your-api-key-here` 时直接抛错提示配置；
  - `OPENAI_BASE_URL`：默认 `https://api.openai.com/v1`，便于接兼容网关/私有化部署；
  - `OPENAI_MODEL`：默认 `gpt-4o`；
  - 调用参数：`temperature: 0.1`（低随机，保证规则稳定）、`response_format: { type: 'json_object' }`（强制结构化输出）。
- **Prompt 设计思路**：
  - 先定义**标准目标字段**（externalCode / storeName / receiverName / receiverPhone / receiverAddress / skuCode / skuName / quantity / spec / remark）及必填性；
  - 给出 `ParseRuleConfig` 的 TypeScript 接口作为"输出契约"；
  - 说明三种解析模式（tabular / card / text）及适用场景；
  - 单列"特殊结构"处理：矩阵转置、尾部信息提取（收货人在数据区外）、多 Sheet 合并、按 `externalCode` 跨行聚合、卡片式堆叠；
  - 约束：只允许输出 JSON、字段名必须用标准名、不硬编码文件名/列名、不确定映射标注 `confidence: low/medium`。
- **健壮性处理**：
  - 解析返回内容先 `JSON.parse`；失败再用正则从 ```json 代码块中提取；再失败抛"AI 生成的规则格式无效"；
  - 从 `fieldMappings` 提取每个字段的 `confidence`，对 low/medium 在 UI 标注"AI 推测"，提示人工复核。
- **设计要点小结**：**结构化输出 + 低温度 + 契约化 Prompt + 容错解析 + 置信度透出**，这是把 LLM 用于"规则/配置生成"类任务的可复用范式。

### 3.2 大模型在 V3 的扩展调用思路（设计建议）

V3 当前未直接调用大模型（FINAL_REPORT 将其列为优化项"集成大模型辅助品控"）。建议按如下思路引入，复用 V2 `ai.ts` 的成熟范式：

#### 3.2.1 适用场景（按价值排序）
1. **品控异常辅助判定**：扫描数据 + 运单明细 + 品控规则结果送入 LLM，生成"异常分类建议 + 处置建议 + 风险等级"，供品控主管快速复核（不替代规则引擎，只做辅助与解释）。
2. **工单智能摘要与自动归类**：对 `ExceptionTicket.description` 做摘要，并建议异常子类型（category），降低录入负担、统一口径。
3. **审批辅助建议**：结合历史审批记录，对工单生成"建议通过/驳回 + 依据"，辅助审批人决策（最终仍由人审批，避免责任主体不清）。
4. **无 V2 时的订单文本兜底解析**：当 V2 不可达且本地无快照，可借鉴 V2 的 `ai.ts`，用 LLM 直接解析用户粘贴的订单文本生成快照（需谨慎，标注"AI 生成，待 V2 复核"）。

#### 3.2.2 推荐实现模式（Server-side only）
- **位置**：新建 `src/lib/ai.ts`（与 V2 同构），仅限服务端（API Route / Server Action）调用，**密钥绝不进前端**。
- **统一封装**：

  ```ts
  // 伪代码思路
  async function callLLM(opts: {
    system: string
    user: string
    temperature?: number
    jsonMode?: boolean
  }): Promise<string> {
    const client = new OpenAI({
      apiKey: process.env.OPENAI_API_KEY!,
      baseURL: process.env.OPENAI_BASE_URL || 'https://api.openai.com/v1',
    })
    const res = await client.chat.completions.create({
      model: process.env.OPENAI_MODEL || 'gpt-4o',
      temperature: opts.temperature ?? 0.2,
      response_format: opts.jsonMode ? { type: 'json_object' } : undefined,
      messages: [
        { role: 'system', content: opts.system },
        { role: 'user', content: opts.user },
      ],
    })
    return res.choices[0]?.message?.content ?? ''
  }
  ```

- **结构化输出契约**：所有 V3 场景都用 `response_format: json_object` + 明确 JSON Schema 文案，返回后 `JSON.parse` 并做字段缺失兜底（与 V2 一致）。
- **可观测与降级**：
  - 每次 LLM 调用写 `SyncLog` 同类日志（模型/耗时/token/错误），与跨系统调用同源可追踪；
  - LLM 超时/不可用**不能阻断主流程**：品控/审批仍按规则引擎与人工进行，LLM 仅作"增强"，失败则静默降级（不展示建议），绝不影响业务判定；
  - 控制成本：品控/审批建议类可缓存（同一工单 `description` 哈希 → 缓存结果），避免重复花费。

#### 3.2.3 提示词（Prompt）工程要点（可复用）
- **给角色与边界**：明确"你是品控/审批辅助，只提供建议，最终责任在人"；
- **给上下文契约**：把运单/扫描/规则结果以紧凑 JSON 形式作为 `user` 内容，避免长文本噪声；
- **给输出 Schema**：用"只输出如下 JSON：{category, suggestion, risk}"约束；
- **给安全护栏**：禁止模型编造运单状态、禁止输出与审批结论相反却无依据的断言；PII（电话/地址）在送入模型前做脱敏，返回后再映射回。

#### 3.2.4 风险与治理
- **责任边界**：LLM 输出一律标注"AI 辅助建议，仅供参考"，关键动作（审批通过/驳回、赔付）必须由人/规则引擎定夺；
- **数据合规**：运单含个人信息，调用外部 LLM 需评估合规；建议支持 `OPENAI_BASE_URL` 指向企业私有网关/国产化模型，密钥与数据不出域；
- **一致性**：LLM 建议与既有 `QCRuleEngine` / `ApprovalEngine` 结论冲突时，以**规则引擎/人工**为准，LLM 仅作为解释性补充；
- **成本与限流**：设单次调用 token 上限与超时（建议 ≤8s），超限即降级。

### 3.3 小结
- **已落地**：V2 用大模型做"出库单解析规则生成"，采用"结构化输出 + 低温度 + 契约 Prompt + 容错解析 + 置信度透出"的稳健范式。
- **待扩展（V3）**：把同一范式复用到品控辅助判定、工单归类、审批建议等**辅助性、非决定性**场景，严格遵守"服务端调用、密钥不下发、失败可降级、责任在人"的治理原则。
